Semantic search is AssetVault's primary search mode. Unlike filename or tag search, it understands **meaning** — you can describe what you're looking for in plain English and the engine finds visually similar assets even if they have no matching tags or filenames.

## How it works

When you type a query:

1. The query string is sent to the Python sidecar.
2. The **query expander** (`query_expander.py`) optionally augments the query with synonyms and related design terms (can be disabled in settings).
3. The CLIP **text encoder** converts the expanded query to a 512-dimensional vector.
4. FAISS performs **cosine-similarity nearest-neighbour search** across all indexed assets.
5. The top-200 candidates are passed to the **multi-signal ranker**.
6. Ranked results (top-50 by default) are returned to Rust, which fetches full asset rows from SQLite and returns them to the UI.

## Multi-signal ranking

The ranker combines four signals into a single score:

| Signal | Weight | Description |
|---|---|---|
| Semantic similarity | 0.6 | Cosine similarity from FAISS (primary signal) |
| Recency | 0.15 | Log-decay bonus for recently modified/viewed assets |
| Favourites | 0.15 | Constant bonus for files you've starred |
| Folder proximity | 0.10 | Small bonus for assets in your most-used folders |

Weights are applied as a weighted sum. The formula is:

```
final_score = 0.6 × semantic + 0.15 × recency + 0.15 × fav_bonus + 0.10 × folder_bonus
```

This means a highly relevant but older asset still outranks a slightly relevant but recently viewed one.

## Query syntax

Semantic search accepts plain-language queries. A few conventions improve precision:

| Query style | Example | Effect |
|---|---|---|
| Descriptive phrase | `dark dashboard with charts` | Best for style/scene matching |
| Object + adjective | `blue rounded button` | Good for UI component search |
| Domain prefix | `figma: hero section` | Boosts design-file results |
| Negative modifier | `icon without background` | Uses CLIP's text-not-image logic |

<Callout type="tip">
  Queries longer than 10 words are automatically truncated by the intent parser
  to avoid degrading CLIP encoding quality. Use the most descriptive 5–8 words
  for best results.
</Callout>

## Query intent parsing

The `intent_parser.py` module parses structured filters from natural language before the CLIP encoder sees the query. Examples:

```
"svg icons created last week"
→ { type_filter: "svg", date_range: "last_7_days", query: "icons" }

"large photos wider than 4K"
→ { min_width: 3840, file_class: "raster", query: "photos" }

"red color palette"
→ { palette_mode: true, color_hint: [255, 0, 0], query: "color palette" }
```

Filters are passed to the Rust layer, which applies them as SQL `WHERE` clauses before the FAISS search to reduce the candidate set.

## Image query (reverse image search)

You can drag and drop an image into the search bar to search by visual similarity rather than text. The image is sent to the sidecar, encoded with CLIP's **image encoder**, and the resulting vector is used as the query. See [Image Search](/docs/image-search) for details.

## Search suggestions

As you type, AssetVault shows query suggestions from three sources:

1. **Rewritten queries** — CLIP-optimised rewrites generated by `query_rewriter.py`
2. **Search history** — your recent successful searches
3. **Tag autocomplete** — matching tags from your indexed library

Suggestions are ranked by frequency and recency and can be dismissed per-session in **Settings**.

## Performance

Semantic search latency is dominated by the CLIP text encoding step on the first query. After the first query, the model stays in memory:

| Step | Cold start | Warm |
|---|---|---|
| CLIP text encode | ~80 ms | ~12 ms |
| FAISS search (50 K assets) | ~3 ms | ~3 ms |
| Re-rank + SQLite fetch | ~5 ms | ~5 ms |
| **Total (UI to results)** | ~100 ms | ~25 ms |

<Callout type="info">
  Search latency is measured on an M1 Pro with 50 000 indexed assets. Larger
  libraries add negligible latency for FAISS queries but more time for SQLite
  row fetches.
</Callout>
